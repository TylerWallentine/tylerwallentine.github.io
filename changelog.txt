FIX/CONSISTENCY: Mobile header + page titles unified across pages.
  Background: styles-phone.css was loaded ONLY on Blog + Main. It overrode

  Result: the raindrop/login bar is now identical on every page (mobile +

FIX: Publishing pipeline — updates weren't reaching GitHub / the live site.
  Diagnosis:
    - All v2 work was committed on local branch "v2-revamp", which had NO
      upstream, so `git push` in update.bat silently failed — but the script
      printed "Done!" unconditionally.
    - origin only had "main" (at old commit dc0be72), and GitHub Pages builds
      from main, so the live site was stuck on the pre-v2 version.
  Fix:
    - Fast-forwarded main to include all v2-revamp commits (dc0be72..46b1483)
      and pushed main to origin — site now publishes the v2 work.
    - Rewrote update.bat: warns if not on main, runs pull --rebase/push with
      explicit branch + upstream (-u), and only reports success if the push
      actually succeeds (fails loudly otherwise).
    - .gitignore: ignore __pycache__/ and *.pyc; removed the accidentally
      committed preview-server .pyc from the repo.
  Going forward: do your work on the "main" branch so update.bat publishes it.

FIX: Blog post viewer — mobile header inset + squished images.
  (1) Header not flush on mobile:
      - styles-phone.css .post-viewer-container had padding: 15px 10px 60px,
        but that container holds the sticky header, so the padding pushed the
        header (and close button) away from the screen edges.
      - Set container padding to 0 and moved inner spacing to a new mobile
        .post-viewer-content rule (padding: 20px 15px 60px; font-size 17px).
        Header now sits flush to the top/sides.
  (2) Images squished:
      - The editor (j-posting.js ~L469-470) writes inline width/height in px
        on resized images. In the viewer, max-width:100% shrank the width but
        the inline height stayed fixed (stylesheet height:auto lost to inline),
        distorting the image on narrow screens.
      - styles.css .post-full-content img: height: auto !important so it
        overrides the inline height and images scale proportionally to the
        container width. (Editor's own resizing is unaffected — rule is
        scoped to .post-full-content.)

FIX/CONSISTENCY: Top header bar + nav now consistent and no longer jitter.

    - The inlined account link defaults to "Log In / Sign Up"; j-firebase's

    - Added theme.css + modern.css to h-blog.html and h-aboutme.html so all
      subpages share the same header/nav styling.
  Notes:
    - modules/account-bar.html and modules/small-grid-container.html are no

    - h-login/h-profile/h-project-editor still use the module loader (login

PERF/FIX: Header background image reloaded/flashed on every page load.
  Causes:
    - images/rains.png was 4.67 MB (1597x1135) — huge for a short header
      banner, so it took real time to fetch/decode on each navigation.
    - preview-server.py sent no-cache on EVERYTHING (incl. images), so the
      image re-downloaded on every local page load (main source of the flash).
  Changes:
    - Added images/rains-header.webp: a cropped/compressed banner
      (1600x500, ~124 KB — 37x smaller than the PNG). Original rains.png
      kept untouched.
    - styles.css: .hero and .hero-small now use rains-header.webp.
    - preview-server.py: static assets (images/fonts/media, by extension)

    - Added <link rel="preload" as="image" href="images/rains-header.webp">
      to h-main, h-projects, h-blog, h-aboutme, h-contact so the header
      image is fetched early.
  Result: after the first load the header image is cached and no longer

FEATURE: Infinite-scroll (lazy-loading) Recent Posts feed on the blog.
  Goal: with 20+ posts, don't fetch them all at once — load a page at a
  time as the reader scrolls.

  How it works:
    - Posts load in pages of RECENT_PAGE_SIZE (6) using a Firestore cursor
      (startAfter) so each page continues where the last left off.
    - An IntersectionObserver watches a 1px sentinel at the bottom of the
      list and fetches the next page ~400px before it enters view.
    - Cards are appended (not re-rendered); a "Loading more posts…"
      indicator shows during fetch and "You've reached the end." when done.
    - Featured post is still skipped in the feed so it isn't duplicated.

  Files changed:
    - j-firebase.js: import Firestore startAfter and expose it as
      window.fsStartAfter (needed for pagination cursors).
      NOTE: initial attempt only added the export line, not the import,
      causing "ReferenceError: startAfter is not defined" at load — fixed
      by adding startAfter to the firestore import list (line 12).
    - h-blog.html: replaced the one-shot limit(6) recent-posts load with a
      paginated feed. Refactored card HTML into renderRecentCard(); added
      window.initRecentFeed() + window.loadMoreRecentPosts() with the
      observer, loading/end states, and a safeguard that keeps loading if
      the first page doesn't fill the viewport (only while visible).
    - styles.css: added #recent-cards (flex column, 25px gap) so the new
      cards wrapper keeps the same spacing as .recent-posts.

  Notes:
    - Uses the existing published+createdAt query, so no new Firestore index
      is required.
    - Search/tag/category filters render into a separate #filtered-posts
      container and hide the recent section, so they don't conflict with the
      feed. Clearing search re-initializes the feed fresh.
